# Text-to-SQL 项目客户访谈要点

## 核心会议目标

1. **对齐需求与边界**：明确 PoC 的业务目标、成功标准以及本阶段“不做”的事情（如暂不涉及复杂的跨库多表关联查询）。
2. **摸底数据现状**：探查客户真实数据库环境、表结构规模及数据质量，夯实 Text-to-SQL 的物理基础。
3. **管理预期与建立信任**：打破“AI全知全能”的错觉，建立合理的验收基线，并彻底打消客户的数据安全顾虑。

---

## 第一部分：会议沟通策略与预期管理 (Expectation Management)

在向客户要数据之前，必须先在认知层面达成共识：

- **锚定 AI 定位（降预期）**：明确告知业务方，第一阶段的 AI 助手相当于一个**“刚入职的初级数据分析师”**。它能极其高效地执行指令明确的取数任务，但暂时无法处理高度模糊的战略性提问（例如不能直接问“为什么上季度利润下降？”）。
- **强调数据安全闭环（建信任）**：主动回应客户痛点。强调所有查询和数据流转都在企业内网闭环，大模型**只处理元数据（如表结构 DDL、字段说明），绝对不触碰、不上传底层核心敏感业务数据**。
- **共创成功标准 / SLA（定基线）**：不要向客户承诺 100% 的准确率。在会议上直接约定 PoC 的及格线（例如：双方确认后，如果 SQL 生成的语法正确率达到 90%，业务语义准确率达到 80%，即视为本次 PoC 验证成功）。

---

## 第二部分：需要向客户提问的问题 (Questions to Ask)

### 1. 数据库与技术架构选型

- **底层数据库类型**：真实的生产或数仓环境使用的是什么数据库？（MySQL, PostgreSQL, Oracle 还是 ClickHouse 等？这直接影响大模型生成 SQL 方言的逻辑）。
- **模型与基础设施**：针对私有化部署要求，目前内网可用的算力资源如何？是否有已经部署好的开源大模型（如特定参数的 Qwen、DeepSeek 等），还是需要我们协助搭建环境？

### 2. 业务场景与数据探查

- **覆盖范围**：本次 PoC 期望覆盖多少张核心业务表？表结构是否规范？（提前预警：若表数量过多或关联过于复杂，会影响大模型的理解能力）。
- **指标计算逻辑**：业务上常用的复杂指标，是已经在底层算好存入结果表了，还是需要大模型当场写复杂 SQL 去聚合计算？（_建议：尽量引导客户在底层建好视图或宽表_）。
  - 样例：客户需要根据“第三季度营业收入”（`rev_q3`）和“第三季度利润”（`profit_q3`）计算“第三季度毛利率”（`margin_q3`）。

---

## 第三部分：需要问客户索要的数据与物料 (Data & Artifacts to Request)

AI 无法凭空理解复杂的企业财报和基金术语，必须向业务方索要以下“弹药”来完成数据理解和模型调优的闭环：

### 1. 详尽的数据字典 (Data Dictionary) **[重中之重]**

- 必须包含表名、字段名、字段数据类型以及**详尽的中文业务注释**。
- 例如：如果数据库字段叫 `rev_q3`，模型必须通过数据字典知道它代表“第三季度营业收入”。

### 2. 业务术语与黑话映射表 (Glossary)

- 机构内部特有的业务方言、别名或缩写。
- 例如：业务方口中的“AUM”或“星辰基金”，具体对应数据库里的哪个准确字段或实体名称？这将用于构建 Prompt 或外挂知识库（RAG）。

### 3. 真实的 DDL 与脱敏样本数据 (Sample Data)

- 提供目标表的 `CREATE TABLE` 语句。
- 每张表提供 **10 - 20 行**真实格式的脱敏数据（务必剥离真实姓名、精确金额等机密）。这能极大帮助大模型理解数据形态（如日期是 `YYYY-MM` 还是时间戳），验证生成的 SQL 是否能真正跑通。

### 4. 黄金测试问题集 (Golden Queries)

- **核心要求**：请业务方不要在未来的演示阶段随意提问，而是现在就写下日常最常问的 **30 - 50 个问题**。
- **交付格式**：“自然语言问题 + 对应的标准 SQL 语句 + 期望的数据结果”。
- **用途**：这套数据集前期将作为 AI 熟悉业务方言的“训练教材”，后期将作为 UAT 验收的“标准考卷”，用来量化我们之前约定的 SLA 准确率。

### 5. 测试环境与账号 (Credentials)

- 数据库的纯只读 (Read-Only) 测试账号及连接方式。
